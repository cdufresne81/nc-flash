# Changelog

All notable changes to NC Flash are documented here.

## [Unreleased]

### Added
- **WiCAN trip logs auto-download into a managed local `logs/` folder (#83)** — NC Flash now pulls trip datalog CSVs off the WiCAN PRO's SD card into `{workspace}/logs`: automatically a few seconds after every app launch (new setting *Settings > ECU > WiCAN > "Auto-download new trip logs on startup"*, default ON — a complete quiet no-op when no WiCAN is configured or the device is asleep/unreachable), and on demand via a new **Download Logs** quick-action in the ECU Programming window. The sync is incremental ("new only": a log already present locally with the same name+size is skipped), skips the still-being-written active trip file (it's picked up next run once the trip closes), and **never deletes anything from the device**. Downloads are atomic — streamed to a `.part` file, verified against the size the device advertised, then renamed into place — so an interrupted WiFi transfer can never leave a truncated CSV that looks complete; device-supplied filenames are sanitized host-side, and clockless-device name collisions (`unknown_time_*.csv` reused across reboots) land under a `-2`/`-3` suffix instead of clobbering. This is a pure HTTP device utility, fully decoupled from the CAN bus/ECU: it needs no ECU connection, works whichever adapter (J2534/WiCAN) is selected and whether the car is on or off, and runs entirely on a background thread (per-file results in the Activity Log; a missing device is a quiet info line, never a dialog). Hardware-validated against the bench WiCAN: 12 real trip logs (7.8 MB) synced byte-identical to independent fetches, re-run fully incremental. New shared transport layer `src/ecu/wican_http.py` (atomic download + JSON GET + basename sanitization) is the single home for WiCAN HTTP plumbing — the upcoming SD file browser (#84) builds on it rather than re-implementing. Also documents a firmware-contract correction: the CSV download endpoint is `GET /download_csv` (issue #83's body says `/csv_download`, which 404s). The whole feature is active only while the WiCAN adapter is selected (with Tactrix/J2534 the SD card is a manual operation: the **Download Logs** button is hidden and the launch auto-sync is dormant), and starting a WiCAN ECU operation first stops any running sync so the download never competes with a flash/read for the adapter's SD/CPU/WiFi. The Settings > ECU > WiCAN page is split by a new "Trip Logs" section header (a `section` descriptor type: rule + header, excluded from settings search) separating the connection settings from the trip-log settings, with Test Connection moved up into the connection group. Tests: `tests/test_ecu_wican_logs.py` (40), `tests/test_wican_log_sync.py` (5, real-QThread lifecycle), `tests/test_ecu_window_download_logs.py` (4, button gating). New settings: trip logs directory (`{workspace}/logs` default) + auto-download toggle, both in Settings > ECU > WiCAN.

### Changed
- **ECU actions "Scan RAM" button relabeled "Read RAM"** — matches the "Read ROM" wording and what the operation does (dumps ECU RAM to a file). Label only; the RAM-read operation is unchanged.
- **Auto-saved ECU reads are named by the ROM's own calibration ID, in all caps** — a full ROM read landed as `ecu_read_<timestamp>.bin` whenever the ECU status card was still unpopulated (it reads `N/A`/`—` before a full read), which is the common case on a fresh read. The read is now saved as `<CAL-ID>_<timestamp>.bin` (e.g. `LF9VEB_20260709_...`), taking the cal ID straight from the bytes just read (`get_cal_id`) — the authoritative name — and falling back to the card, then the generic `ecu_read`, only when the bytes carry no valid cal ID. A RAM dump (which carries no cal ID) reuses the status card's ROM_ID and lands as `<ROM_ID>_RAM_<timestamp>.bin`. The ROM_ID is spelled in all caps in both filenames (the generic `ecu_read` fallback stays lowercase). Tests: `tests/test_ecu_read_naming.py`.
- **Flash-complete message no longer overstates the ignition cycle** — the post-flash dialog said you *MUST* cycle the ignition (key OFF ~10 s, then ON) and that the ECU "stays in its programming bootloader until you do". The ECU reboots on its own after a valid flash (its ECUReset brings up the new calibration), so cycling the key is not required — it stays recommended as a way to confirm the new calibration booted cleanly. The completion dialog, its progress line, and the pre-flash WiCAN confirmation now say the ECU reboots itself and frame the ignition cycle as "not required but recommended" (the specific ~10 s wait is dropped). The completion dialog also drops its "verify the flash byte-for-byte, use Read ROM…" line (an inline read-back isn't possible right after a flash). Message/wording only — the flash sequence is unchanged.

(2) `tests/test_flash_prep_equivalence.py` gained a static composition ratchet proving both flash modules call `prepare_flash_image` and neither references `correct_rom_checksums`/`get_sbl_data` directly — pinning the J2534 side, which the byte-equality tests alone could not (both sides composed the same function). (3) `tests/test_theme_ratchet.py` now scans `main.py` (MainWindow, the largest widget, was outside the ratchet with 4 literals) and the ECU window's new disconnect-amber moved into `theme.WARNING_AMBER` (budget 40→39). (4) Doc refresh: `WICAN_TRANSPORT.md` §6 no longer describes the retired host-driven WiCAN write as the implemented path (now documents the SD-staged flash); `ARCHITECTURE.md` no longer points at untracked `.claude/plans/` files as canonical evidence and names the real theme builder; three `ecu_window.py` docstrings and `tools/wican_flash_diag.py`'s usage/help no longer advertise the retired `WiCANFlasher` flash / `--commit` real-flash invocation. (5) The new B3 per-ROM path-normalization test is Windows-guarded — it asserts the backslash/forward-slash path unification that `pathlib` provides only on `nt` (the app's deployment target), so it was failing the Linux CI legs while passing on Windows; the production filter is unchanged and correct on Windows.
- **Theme module — one home for widget colors + a shared toolbar stylesheet (F1/F2 foundation)** — Added `src/ui/theme.py` (named color constants + a `get_toolbar_stylesheet(checked)` builder), the flat "constants + builders" source of truth the app's force-Light styling was missing. The identical flat-toolbar QSS that was pasted into three windows (main, table viewer, compare) now calls the builder — `table_viewer_window.py` dropped to zero raw color literals. A shrink-only ratchet (`tests/test_theme_ratchet.py`) budgets raw color literals per widget file (baseline 104, theme.py exempt) and fails if any file grows or a new one adds inline hex — so the remaining `ecu_window`/`settings_dialog`/`log_console` migrations can only shrink. Not a framework, not dark mode.
- **Quieter, more diagnosable ECU logging (G5/G6, pure logging — no wire change)** — On the no-reboot coexist port, interloping datalogger OBD replies (`0x41`) that land while the host holds the bus are benign noise, not faults; the per-frame `WARNING` is demoted to `DEBUG` (G5). And `read_engine_rpm`/`read_battery_voltage`, which swallow every exception to stay fail-open, now log the swallowed cause at `DEBUG` so a real comms fault isn't misread as "PID unsupported" in a post-mortem (G6). Fail-open behaviour unchanged.
- **ROM detector warns on ambiguous definitions (H10)** — When two ROM definitions share an `internalidstring`, detection silently returned whichever the glob loaded first. It now collects all matches, logs a `WARNING` naming them when there is more than one, and still uses the first (behaviour preserved).
- **CI now tests the interpreter development runs on (H2)** — Added a `windows-latest / Python 3.14` leg to the CI matrix (dev happens on Windows + 3.14.2, which no CI job previously exercised), alongside the existing 3.10/3.12 legs.
- **README documents WiCAN wireless flashing (H1)** — The Overview and ECU Flashing feature list now mention reading/flashing over a WiCAN PRO adapter (SD-staged, firmware-driven, datalogger-coexisting; adapter picked in Settings > ECU > Adapter, mDNS auto-discovery) — previously absent despite being a major shipped capability.
- **Session logs are pruned to the newest 30 (H5)** — one per-session log is written per launch to `~/.nc-flash/logs`; the directory no longer grows without bound (best-effort prune at startup, never fatal).
- **Brick-critical flash-prep is now ONE pipeline shared by both flash paths (D1)** — The host-side prep that turns a ROM into flashable bytes — validate → detect vehicle generation → checksum-correct (+ verify zero residual) → bounds-check → build the matching secondary bootloader (SBL) — was copied step-for-step in `FlashManager._flash_rom_inner` (the J2534 wired flash) and `wican_sd_package.build_flash_package` (the SD-staged wireless flash). Its own docstring said it "replicates ... exactly", but nothing enforced that — and any drift (a mismatched SBL, an uncorrected checksum) bricks an ECU. Both now compose a single pure `src/ecu/flash_prep.py::prepare_flash_image(rom_data, flash_start_index)`. A new equivalence gate (`tests/test_flash_prep_equivalence.py`) proves (a) the function reproduces the known-good prep byte-for-byte against an independent recompute and (b) the two flash paths emit identical corrected-ROM + SBL + program bytes. Behavior-preserving on the success path (the ECU sees the exact same command sequence and bytes). Failure-path deviations, all pre-ECU-contact and fail-fast: the SBL is now computed pre-connect, so an unsupported start index fails BEFORE any ECU contact instead of mid-session and surfaces as a typed `FlashError` instead of a raw `ValueError`; in `build_flash_package` the identical-ROMs check now precedes checksum errors; a residual-checksum failure's `ChecksumError` message now carries the applied-corrections count and the residual regions (previously that diagnostic was lost because the corrections list only reached the caller's log on success). The next bench SD flash doubles as belt-and-braces.
- **Compare/table render + format/DTC dedup (D3 step 1, D7, D8, D9)** — Removed several verbatim duplications the audit flagged: (D3) the value→ratio→color math (`if max==min: 0.5 else clamp(...)`) that the compare window and the table display each re-implemented now lives once in `utils/colormap` (`clamp_ratio` / `value_to_color`); (D7) the table display's private `get_value_format`/`_get_axis_format` were re-deriving what `utils/formatting.get_scaling_format`/`get_axis_format` already provide (and the compare window already used) — deleted, imported shared; (D8) the DTC de-duplication one-liner duplicated across two `ecu_window` sites is now a single `src/ecu/dtc.py::dedup_dtcs`; (D9) `flash_manager` repeated `from .constants import ISO15765_BS, ISO15765_STMIN` inside five methods — hoisted to the module import. All behavior-preserving.
- **matplotlib is no longer imported at startup (E2)** — It was pulled in eagerly via `main → table_viewer_window → graph_viewer`, costing ~1.15 s of measured cold-start import time even though the only consumer (`GraphWidget`) is built lazily — only when a 2D/3D table is opened. The `Figure`/`FigureCanvas` imports moved into `GraphWidget.__init__`; a subprocess ratchet (`tests/test_lazy_matplotlib.py`) fails if matplotlib re-enters the startup import graph. Startup is ~1.15 s faster; opening the first graph pays that cost once.
- **Per-ROM edit state now has ONE owner — `RomDocument` (C1/C2/C3/C5; behavior-preserving, adversarially verified)** — The root cause of nearly all the audit's UI debt: `modified_cells`, `original_table_values`, and `rom_colors` lived as `MainWindow` dicts keyed by `rom_path` and were handed **by reference** into each `TableViewer`, so both the viewer and `MainWindow` mutated the same dict object — the single-owner-state rule's motivating incident. That state now lives on each `RomDocument`: a new pure `src/core/table_edit_state.py` `TableEditState` (modified-cell borders + capture-once, deep-copied original snapshots) owned as `document.edit_state`, plus the per-ROM tint as `document.get_color()/set_color()`. `TableViewer` takes an `edit_owner` (the document's `TableEditState`, or a private throwaway when standalone) and *delegates* all border storage to it — no dict crosses an object boundary, and the 1e-10 smart-border-removal algorithm stays in the viewer. **(C2/C5)** `TableViewerWindow` now re-emits its viewer's edit signals as `cell_edited`/`bulk_edited`/`axis_edited`/`axis_bulk_edited` with the window's `rom_path` bound, so `MainWindow`'s handlers always know which ROM was edited; the `_get_sender_rom_context()` `sender().parent()`-walk **and its silent fallback to the active tab** are deleted — an edit whose `rom_path` can't be resolved now fails loud (logs + skips) instead of writing to whatever tab happens to be focused. The color allocator, tab swatch, picker, compare, close, and Save-As paths all go through the document; `MainWindow` keeps only the palette-cycle allocator state. **(C3, minimal)** committing a project now re-baselines the committed ROM's edit state (`_reset_document_edit_baseline`): modified-cell borders clear and originals re-capture from the committed bytes (previously borders lingered until the tab was closed or Save-As'd), while undo stacks stay intact so a committed change can still be undone. Five independent adversarial skeptics (edit attribution, border/original semantics, color allocator, lifecycle/leaks, commit re-baseline) confirmed no behavior regression. Tests: `tests/test_table_edit_state.py`, `tests/test_phase3_edit_ownership.py` (+ updated `test_edit_write_commit`, `test_commit_scope`). Part of `.claude/plans/architecture-goal.md` Phase 3.

### Fixed
- **Interleaved 3D tables no longer render a garbage grid when the definition address is stale** — The interleaved reader trusted the two bytes at the table address as the `[M][N]` dimension header unconditionally; a definition whose address doesn't point at the real header (user report: an LF9KT xml ported from the LFG1T definition without re-basing addresses — the LF9KT firmware shifted the same data blocks by −4/+72/+216 bytes) made arbitrary data bytes act as dimensions, rendering e.g. a 5×3 table as an 80×64 grid of garbage ("hundreds of cells"). The ROM header is now cross-validated against the definition's declared axis element counts in ONE helper (`RomReader._interleaved_dims`) used by all four interleaved paths (read, bulk write, cell write, axis write); a mismatch raises a descriptive `RomReadError`/`RomWriteError` naming both dimension pairs and the likely cause, which the UI already surfaces as an error dialog. Definitions without axis children (or with elements-less children) stay self-describing (unchanged). Also documented the previously-undocumented `layout="interleaved"` format (structure, address-must-point-at-header rule, validation) in `docs/internal/ROM_DEFINITION_FORMAT.md`. Verified against the reporter's ROM/XML pair: 18 stale defs now fail loudly instead of silently rendering garbage, and all 100 interleaved tables in the shipped LFG1TF000 example pair still load. An adversarial review workflow (5 agents) confirmed the diagnosis (alternative reader-bug/parser-bug explanations empirically refuted: the reader's header convention fits 100/100 working defs, swapped/off-by-one conventions fit 0/66) and surfaced three additional hardening fixes shipped with this change: **(a)** interleaved X-axis writes now derive their address from the validated table base instead of the child axis address — a stale child address (present on the reporter's one "working" table) would have silently redirected an X-axis breakpoint edit into a Y-axis byte; **(b)** `swapxy` + interleaved is now rejected outright (read reshapes C-order, write flattened F-order — a round-trip would silently transpose the grid; no shipped def uses the combination); **(c)** a partial-axis mismatch message renders the undeclared dimension as `?` instead of `None`. Tests: `tests/test_interleaved_tables.py` (+15, including single-dimension-mismatch mutants, write-path rollback assertions, and the stale-child-address X-write regression).
- **Bulk and external edits now honor the write-then-notify commit point (B8 completion)** — B8 converted the single-cell/axis handlers to write-first, but the bulk handlers (Interpolate/Multiply/Set), the compare-copy pipeline, and MCP AI edits still recorded undo + pending BEFORE the ROM write and ignored a failed write — a mid-loop `RomWriteError` left the UI/undo/pending asserting values the ROM never fully took (and MCP reported `success: true`). All four paths now write first; on failure they roll back the cells that did land (best-effort re-write of the old raw values), revert the viewer, record nothing, and surface the error — compare-copy shows a dialog, MCP returns `success: false`. Tests: `tests/test_edit_write_commit.py` (+11).
- **A hung NC Flash no longer makes a relaunch exit silently (B11 hardening)** — the single-instance gate exited 0 purely on a successful socket *connect*, but the OS accepts a QLocalServer connection even when the running instance's GUI thread is deadlocked — double-clicking the app then did nothing, with Task Manager the only recovery. The running instance now ACKs the handoff from its event loop; no ACK within 2 s → the new launch opens a working window (the pre-B11 recovery path, now only when actually needed). Tests: `tests/test_single_instance.py`.
- **A failed project open no longer blocks every later project open (B4 hardening)** — a project that bound the `ProjectManager` but failed to produce a tab (definition XML missing/renamed, load error) left the manager bound to an invisible project; the one-project guard then silently skipped every subsequent open, including session restore of a healthy project. The failure paths now unbind the tabless project, and a session-restore skip is surfaced in the status bar instead of vanishing silently. Tests: `tests/test_project_switch_guard.py` (+3).
- **A malformed axis no longer leaves its parent table crippled (B10 completion)** — the B10 numeric guard silently dropped just the failing X/Y-axis child, so the parent 2D/3D table survived as a flat value list without breakpoints. A parent whose axis child fails to parse is now skipped whole (logged), matching the guard's "skip just that table" contract. Tests: `tests/test_definition_parser.py` (+2).
- **The atomic scaling write cleans up its temp file on failure (B9 completion)** — if `os.replace` failed (e.g. the XML held open by another program on Windows), the `.tmp` serialization was left littering the metadata directory.
- **GUI test runner bulk commands no longer crash (dev tool)** — `set_value`/`multiply_selection`/`add_to_selection` emitted `bulk_changes`/`axis_bulk_changes` with one argument against `Signal(object, list)`, raising `TypeError` on every invocation (pre-existing at HEAD). They now pass the table as the signals require.
- **Reverting a project version now targets the PROJECT's ROM, not the active tab** — `_on_revert_version` resolved the document via `get_current_document()`: with a foreign tab active (e.g. a second ROM open for Compare), the revert reloaded the WRONG document from disk (discarding its unsaved in-memory edits) while the project document kept stale pre-revert bytes that a later save would silently write back — undoing the revert. It also cleared pending changes globally (wiping the foreign ROM's uncommitted edits from the tracker) and left undo stacks recorded against pre-revert bytes, which would write pre-revert values into the reverted ROM if replayed. The fix mirrors the B3 commit scoping: resolve the document by `working_rom_path`, close the ROM's open table windows (they show pre-revert values and have no reload path), drop its undo stacks (as tab close does), clear only its pending changes + dirty flag, and re-baseline its edit state. Tests: `tests/test_revert_scope.py`.
- **External axis edits (compare-copy) now refresh an already-open table window** — `_apply_external_cell_edits` updated any open viewer after writing the ROM, but its axis twin `_apply_external_axis_edits` did not, so copying a table whose axes differed left an open destination window showing stale axis values (the ROM bytes, undo, and borders were already correct). The axis path now mirrors the cell path's viewer refresh via `update_axis_cell_value` (the same primitive undo/redo uses).
- **Dragging a ROM tab no longer misdirects edits/saves/flash to the wrong ROM (B1)** — The tab strip is `setMovable`, but a drag reordered only the `QTabBar`, not the parallel `QStackedWidget`, so after a reorder every index-based lookup (`get_current_document`, close, save, flash) resolved to a *different* ROM than the one whose tab was selected. A new `tabMoved` handler now moves the corresponding stack page in lockstep and re-syncs the current page, so the tab and its ROM stay bound. Tests: `tests/test_tab_reorder.py`.
- **Undo/redo after a save is no longer silently discarded on close (B2)** — Save/commit clear the dirty flag but leave the QUndoStack intact, so undoing a change that was already saved wrote ROM bytes without re-marking the document modified; closing then discarded that undo with no prompt. The undo appliers now set the document modified after a successful write, so the "*" marker and the close/save prompt reflect it. (QUndoStack becomes the single dirty authority in Phase 3.) Tests: `tests/test_undo_dirty_flag.py`.
- **Committing a project no longer folds in other open ROMs' edits or snapshots the wrong tab (B3)** — The ChangeTracker is global across all open ROMs, so a commit snapshotted the *active* tab and wrote *every* open ROM's pending edits into the project history — a second ROM open for Compare would pollute the commit, and the project's own edits could be omitted from the snapshot. Commit now resolves the project's own ROM document (via `working_rom_path`), flushes and snapshots *that* file, commits only its pending changes, and clears only its tracking — foreign tabs keep their uncommitted edits. New `ChangeTracker.get_pending_changes_for_rom` / `clear_pending_for_rom`. Tests: `tests/test_commit_scope.py`.
- **Opening a second project no longer hijacks the first project's tab (B4)** — The `ProjectManager` is a singleton (one `current_project`); opening a different project rebound it while the first project's tab stayed open, so that tab would then commit/revert into the *new* project's history. Opening a different project now prompts to close the current one first (one project at a time); declining aborts, and session restore keeps the already-open project instead of prompting mid-startup. Tests: `tests/test_project_switch_guard.py`.
- **A failed save-on-close no longer crashes the app or loses edits (B5)** — Both close paths (`_handle_close` for the window, `close_tab` for a single tab) called `document.save()` bare; a `RomFileError`/`RomWriteError` there escaped `closeEvent` (a hard crash under PySide6) and the edits vanished with no dialog. The save is now guarded: on failure the close is cancelled and the error is surfaced through the shared error helper, so the user keeps their work. Tests: `tests/test_close_save_guard.py`.
- **A rejected cell/axis edit no longer leaves the UI asserting a value the ROM never took (B8)** — Interactive edits recorded undo + pending *before* writing the ROM, and the write only logged on failure, so a rejected write left the undo stack, pending set, and viewer showing a value the ROM didn't hold (later shipped by Save/flash). The single-cell and single-axis handlers now treat the ROM write as the commit point: they write first and only record undo/pending on success; on failure they revert the viewer cell and surface the error. (Bulk/external edit paths get the same transactional treatment when Phase 3 unifies edit state.) Tests: `tests/test_edit_write_commit.py`.

- **WiCAN read-speed software levers (Phase 1)** — Three behaviour-preserving optimisations toward J2534/Tactrix read parity, all defaulting safe and tunable from the bench. - **WiCAN transport hardware-hardening (read path validated end-to-end)** — Three reliability fixes found and confirmed against a live MX-5 NC ECU over a WiCAN PRO. - **CI: intermittent `ZeroDivisionError` in `FlashManager.read_rom`/`_flash_rom_inner` on fast runners** — The ROM-read and ROM-transfer **completion** log lines computed `bytes / elapsed` without guarding `elapsed == 0`. With a mocked UDS (tests) the full 1 MB read/transfer can finish inside a single `time.monotonic()` tick, so `elapsed` is exactly `0.0` and the summary line raised `ZeroDivisionError` (`src/ecu/flash_manager.py:878`/`:704`) — failing `pytest` non-deterministically depending on runner speed. Tests: `tests/test_ecu_flash_manager.py::TestReadRomBlockSize::test_zero_elapsed_read_does_not_divide_by_zero` (pins `monotonic` to a constant to force the zero-elapsed race deterministically).
- **CI: flaky WiCAN transport test on slow runners** — `test_ecu_wican_transport.py::TestChannelPriming::test_flush_drains_pending_frames` held its mock-server socket open with a fixed `sleep(1.5)` that could expire mid-`receive_message()` on a slow runner, so the client correctly reported "socket closed by peer" and the test failed intermittently. The handler now holds the socket open until the client disconnects, removing the timing dependency. Test-only — no product/behaviour change.
- **DTC Activation Flags toggle switch crashed on every flip — `cell_changed` emitted the table address instead of the Table object** — Flipping a DTC-flag toggle (a 1-D toggle-category table, e.g. `P0222`/`P0122`) raised `AttributeError: 'str' object has no attribute 'address'` in both the modification tracker (`table_viewer._on_cell_changed_track_modifications`) and the undo/redo recorder (`main._on_table_cell_changed → table_undo_manager.record_cell_change`). The toggle handler `_on_toggle_changed` emitted `cell_changed` with `current_table.address` (a `str`), while every other edit path — and every one of the three `cell_changed` consumers — passes/expects the **Table object** and reads `table.address`/`table.name` itself. Pre-existing since the toggle feature shipped (commit `05ebbeb`, 2026-02-07); surfaced now that the consumers always dereference `.address`. Fixed by emitting `self._ctx.current_table`, matching the normal cell-edit emit in `editing.py`. Tests: `tests/test_table_viewer_window.py::TestSignalForwarding::test_toggle_emits_table_object_not_address` drives the real toggle handler (the prior signal-forwarding tests emitted manually, so they never exercised this path).
- **ECU operations crashed / threw cross-thread Qt warnings on completion — `finished`/`error` handlers ran on the worker thread** — `_start_flash` connected `worker.finished`/`worker.error` to **bare lambdas** with `Qt.QueuedConnection`. A bare lambda has no receiver QObject, so Qt ran the slot in the **worker** thread instead of the GUI thread; `_on_flash_finished` then mutated widgets (`setVisible`, the completion `QMessageBox`, repaint) off the GUI thread, emitting `QObject::setParent: Cannot set parent, new parent is in a different thread` and intermittently crashing on a cross-thread paint (`QBackingStore::endPaint() called with active painter`). Seen after a dynamic flash **and** a RAM scan — both of which had actually completed on the ECU; the failure was purely in the post-operation UI. Fixed by connecting **bound methods** of the window (`_on_worker_finished` / `_on_worker_error`): the window is a GUI-thread QObject, so the queued slot is delivered to the GUI thread (the same reason `progress → self._on_flash_progress` was always safe). The handlers read the stored `_flash_thread`/`_flash_worker`. Root-caused by the new Qt diagnostics below — the captured Python stack pointed straight at `_on_flash_finished → _btn_done.setVisible(True)`. Tests: `tests/test_ecu_window_flash_driver.py::TestWorkerFinishedHandlers`.
- **WiCAN ROM read aborted near completion on a brief link stall — more idempotent retries + backoff** — A live 1 MB read over WiCAN failed at block ~957/1024 (offset `0x0EF400`) after only 4 back-to-back `ReadMemoryByAddress` drops at one offset (ISO-TP consecutive-frame N_Cr timeout), discarding ~5 minutes of work to a sub-second transient (WiFi roam / interference burst / momentarily wedged gateway). A single dropped frame already fails one attempt fast by design, but four *consecutive* drops mean a brief stall, not a dead link. `READ_BLOCK_RETRIES` is raised 4 → 8 and each retry is now spaced by a short, growing backoff (`0.2 s × attempt`, capped at 1 s) so the stall clears before re-requesting — only the (rare) failing block ever waits, and never after the final attempt. Reads are idempotent, so extra attempts can never change ECU state; a clean J2534/WiCAN block still returns on attempt 1 with no wait. The flash/write path is untouched (it must never resend a block). Tests: `tests/test_ecu_flash_manager.py::TestReadBlockRetry` (backoff grows between attempts, never after the last; recovery still flushes between attempts).
- **Spurious "Invalid state transition blocked: idle → authenticating" error during WiCAN SD flash** — `WiCANSdFlasher._authenticate_ecu` called `FlashManager._authenticate()` straight from the `IDLE` state, which the state machine rejects (only `CONNECTING → AUTHENTICATING` is valid), logging a red-herring `ERROR` on every SD flash even though auth then proceeded. It now calls `_connect()` first (borrowed mode → Tester Present + `IDLE → CONNECTING`), exactly like the read path, so the transition is valid, the log is clean, and the ECU gets a liveness check before auth. No behaviour change to a successful flash.
- **WiCAN WRITE/flash failed on the first `TransferData` block — outbound consecutive-frame pacing added** — A real FULL FLASH over the WiCAN failed: auth, security, and `RequestDownload` all succeeded, but the very first SBL `TransferData` (SID 0x36) timed out after 60 s ("Timed out waiting for response to SID 0x36"). Root cause (hardware-confirmed): on a write the **tool** sends the 1 KB block as an ISO-TP First Frame + ~146 Consecutive Frames, and the NC ECU's Flow Control advertises **BS=0, STmin=0** ("send everything back-to-back") — so the unpaced burst overruns the WiCAN gateway's TCP→CAN forwarding buffer, a frame is dropped *inside the gateway*, the ECU's reassembly never completes, and it never answers. This is the exact mirror of the receive-side overflow `rx_stmin` already guards (reads work because the ECU is the multi-frame *sender* and the host has N_Cr fast-fail + idempotent retry; writes have no mid-stream resend by design). Fix: `IsoTpSession` gains a `tx_stmin` outbound STmin **floor** applied as `max(peer_stmin, floor)` between our Consecutive Frames; `WiCANTransport` defaults it to **3 ms** (`DEFAULT_TX_STMIN`). It only changes inter-frame *timing* within one message — never the payload, never a resend, never a block split — so the no-mid-stream-resend (anti-brick) invariant is preserved; J2534 and reads are unaffected (default floor 0 / single-frame requests). **Hardware-validated at the block level:** with the 3 ms floor a paced 147-frame SBL block ACKed cleanly in 638 ms on the live MX-5 NC ECU (`tools/wican_flash_diag.py`, pre-erase diagnostic). Full-1 MB flash completion over WiFi (no resend across ~150k frames) is the remaining open validation; SD-staging (Option B) stays the robust long-term path. Tests: `tests/test_ecu_isotp.py::TestOutboundTxStminFloor` (floor paces at STmin=0, peer wins when larger, default 0 unchanged, applies regardless of `honor_peer_fc`, payload byte-identical) + `tests/test_ecu_wican_transport.py` (floor wired into the session). See `docs/internal/WICAN_TRANSPORT.md` §6.
- **WiCAN RAM scan aborted on a single dropped frame** — `FlashManager.scan_ram` read each RAM page with a bare `read_memory_by_address` (no retry), so over the lossy WiCAN/WiFi link one dropped consecutive frame failed the whole 48 KB scan (caught on the real ECU: the scan died at page 53/192 with an N_Cr timeout). It now uses the same idempotent `_read_block_with_retry` as `read_rom` — a dropped page is re-requested (reads never change ECU state), flushing stale frames between attempts. Confirmed on the live MX-5 NC ECU over WiCAN: a full 48 KB RAM dump completes, transparently recovering the occasional dropped page. The J2534 (reliable) path is unaffected — the first attempt always succeeds, so it stays a straight passthrough.
